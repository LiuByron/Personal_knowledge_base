# SDD + Harness 规范驱动开发 + 标准化开发

## SDD：规范驱动开发

**SDD（Spec-Driven Development）** 是一种以规范文档（Spec）为核心驱动力的 AI 辅助开发方法论。它通过预先定义清晰、结构化的技术规范文档，来约束和引导 AI 的代码生成过程。

> **核心理念**：先写 Spec，再写 Code —— 通过结构化的规范文档为 AI 提供精确的上下文和约束条件，确保生成代码的质量、一致性和可维护性。

SDD 的本质是将传统软件工程中的"设计先行"原则与 AI 编程相结合，用文档化的规范取代模糊的口头描述，让 AI 真正理解开发者的意图。

### SDD 核心工作流

1. **需求分析**：明确功能需求与技术约束
2. **Spec 文档编写**：API Spec / 数据模型 / UI 规范 / 测试 Spec
3. **AI 驱动代码生成**：基于 Spec 约束生成高质量代码
4. **Spec Review 与验证**：自动化质量校验与人工审查
5. **迭代优化**：持续完善 Spec，提升输出质量

### SDD 能解决的问题

| 问题 | 解决方式 |
|------|----------|
| **代码质量一致性** | 通过 Spec 规范约束 AI 输出，确保代码风格、架构模式的统一 |
| **需求到代码的精确传递** | 结构化 Spec 消除自然语言歧义，减少 AI"误解" |
| **团队协作标准化** | 统一 Spec 模板让团队用一致方式与 AI 交互 |
| **知识沉淀与复用** | Spec 文档是项目知识的结构化载体 |
| **质量可追溯** | 每段代码有对应 Spec，出问题可追溯源头 |

### SDD 解决不了的问题

- **AI 模型能力上限**：模型本身不支持某种范式，Spec 再好也无能为力
- **创意和直觉决策**：产品方向、UX 设计等需要人类创造力的领域
- **复杂架构决策**：技术选型、系统架构等仍需资深工程师判断

> **SDD 的边界认知**：SDD 是方法论层面的解决方案，规范"如何与 AI 协作"，而非"AI 本身能力"。需配合 Harness 实现工程级落地。

## Harness Engineering 是什么？

**Harness Engineering（驾驭工程）** 是一种为 AI 编程建立自动化约束和质量保障机制的工程方法，它像高速公路的护栏一样，确保 AI 编程在正确的轨道上高速运行。

> **类比理解**：如果 SDD 是"导航系统"（告诉 AI 往哪里走），那么 Harness Engineering 就是"高速公路护栏"（确保 AI 不会偏离轨道）。SDD 定义方向，Harness 保障安全。

Harness Engineering 的核心价值在于：将人工的代码审查、规范检查、质量验收等流程自动化，让 AI 编程的输出始终符合工程标准。

### Harness Engineering 的三大能力

- **自动化约束规则**：代码风格、命名规范、架构约束的自动检查
- **工程护栏机制**：防止 AI 输出偏离预定架构和设计模式
- **质量门禁系统**：自动化测试、静态分析、安全扫描

### Harness Engineering 的六大核心能力

1. **代码规范自动约束**：自动检查代码风格、命名规范、目录结构是否符合团队标准。示例：自动拒绝不符合命名规范的 PR
2. **架构一致性守护**：确保 AI 生成的代码符合预定的架构模式，防止架构腐化。示例：限制跨层调用、强制依赖注入
3. **自动化测试集成**：AI 生成代码后自动触发单元测试、集成测试验证。示例：代码生成后自动运行测试套件
4. **文档体系管理**：结构化的文档目录和模板管理，确保知识沉淀。示例：`.harness` 目录下的规范文件体系
5. **协作流程规范化**：定义团队成员与 AI 协作的标准流程和交互模式。示例：Spec 编写 → Review → 生成 → 验收
6. **质量度量与反馈**：持续收集代码质量数据，优化 Spec 模板和约束规则。示例：自动生成代码质量报告和趋势分析

> **总结**：Harness Engineering 将传统代码审查中"人工判断"的环节自动化，让 AI 编程的质量保障不再依赖于个人经验。

### Harness Engineering：三层护栏

> 给 AI 这匹"烈马"套上缰绳 —— SDD 决定方向，Harness 确保不跑偏。

| 层级 | 名称 | 核心内容 | 具体手段 |
|------|------|----------|----------|
| 第一层 | **约定层（Convention）** | 告诉 AI"我们团队的规矩" | `.cursorrules` + `.kiro/steering/` 技术栈约定 |
| 第二层 | **自动化层（Automation）** | 让机器自动检查 | Pre-commit Hook + ESLint + Prettier + CI 安全扫描 |
| 第三层 | **验证层（Validation）** | 最终确认达标 | 自动化测试 + AI Code Review + 对照 Spec 验收标准逐条检查 |

> **效果**：引入 SDD + Harness 后，合并冲突减少 **60%**，Code Review 时间缩短 **45%**。

## SDD + Harness 与 Vibe Coding 认知误区

### 核心误解澄清：SDD 不是替代 Vibe Coding

| 错误理解 | 正确理解 |
|----------|----------|
| SDD + Harness 的兴起 = Vibe Coding 的终结 | SDD + Harness 是 Vibe Coding 的**进化版本** |
| 我们正在"回归传统工程" | 保留交互灵魂，穿上工程师铠甲 |
| Vibe Coding 已经过时了 | 变的不是交互方式，而是加上了"工程规矩" |

### 五大常见认知误区

| 误区 | 正确理解 |
|------|----------|
| **误区一**："学了 SDD 就不需要 Vibe Coding 了" | Vibe Coding 是基础交互方式，SDD + Harness 是工程化升级，是 **"基础 + 增强"** 关系 |
| **误区二**："SDD 太重了，会降低开发效率" | 初期投入稍高，但长期节省大量调试和返工时间，Spec 复用率可达 **70%+** |
| **误区三**："Harness 只适合大团队" | 个人开发者同样受益于自动化约束，减少低级错误，提升代码一致性 |
| **误区四**："写 Spec 比写代码还费时间" | 好的 Spec 让 AI 一次性生成高质量代码，减少反复修改，总体效率提升 **3-5 倍** |
| **误区五**："有了 AI 就不需要工程规范了" | AI 越强大，规范约束越重要。越快的赛车越需要更好的赛道护栏，防止 AI 偏离轨道 |

## 三者关系：Vibe Coding / SDD / Harness

### 层级关系图解

```
┌─────────────────────────────────────────────┐
│  Harness Engineering — 工程保障层             │
│  自动化约束 | 质量门禁 | 持续验证 | 协作规范    │
│  → 提供工程护栏保障                           │
├─────────────────────────────────────────────┤
│  SDD（Spec-Driven Dev）— 方法论层             │
│  Spec 文档体系 | 需求到代码的规范化映射 | 质量标准定义 │
│  → 提供结构化约束                             │
├─────────────────────────────────────────────┤
│  Vibe Coding — 交互方式层                     │
│  自然语言对话 | AI 理解意图 | 代码生成 | 快速迭代 │
└─────────────────────────────────────────────┘
```

### 类比理解

| 类比维度 | Vibe Coding | SDD | Harness |
|----------|-------------|-----|---------|
| **驾驶类比** | 会开车（基本能力） | 有导航系统（知道往哪走） | 有高速公路护栏（保证安全） |
| **建筑类比** | 工人技能（能干活） | 建筑设计图纸（干什么） | 施工规范标准（怎么干才合格） |
| **写作类比** | 能写字（基础能力） | 文章大纲（结构化思考） | 出版规范（质量保障体系） |

> **核心认知**：三者不是互相替代，而是层层递进的增强关系。Vibe Coding 是基座（提供能力），SDD 是方法论升级（提供方向），Harness 是工程保障（提供保障），**三者缺一不完整**。

## SDD + Harness 如何让 Vibe Coding 走向工程化？

### 从痛点到解决方案

| 痛点 | 解决方案 |
|------|----------|
| **输出质量不稳定** | SDD → Spec 规范约束 AI 输出范围和标准 |
| **缺乏规范约束** | Harness → 自动化护栏检查代码合规性 |
| **团队协作困难** | SDD + Harness → 统一的 Spec 模板 + 协作流程规范 |
| **知识传递断裂** | SDD → Spec 文档即知识沉淀载体 |

- **SDD 提供规范化的输入**：让 AI"知道该做什么、怎么做、做到什么标准"
- **Harness 提供自动化的保障**：让 AI 输出"自动被验证、自动被约束、自动被修正"

> **工程化公式**：SDD（规范输入）+ Harness（自动约束）= 可控的、高质量的 AI 编程体验

SDD + Harness 不是取代 Vibe Coding，而是在保持 AI 编程高效性的同时增加工程化保障，适合需要长期维护的生产级项目。

### 完整方程式与五步工作流

> **工程化 Vibe Coding = 需求对话 → Spec 生成 → 代码实现 → 护栏检查 → 验收对照**

| 步骤 | 名称 | 角色 |
|------|------|------|
| 1 | **需求对话** | Vibe Coding 灵魂 |
| 2 | **Spec 生成** | SDD 核心 |
| 3 | **代码实现** | AI 执行 |
| 4 | **护栏检查** | Harness 保障 |
| 5 | **验收对照** | 质量闭环 |

## AI 模型决策树：如何选择模型

### 决策树第一层：按任务难度切

### 常见反模式：别踩这 5 个坑

| 反模式 | 错误认知 | 正确理解 |
|--------|----------|----------|
| **反模式 1："贵 = 好 / 最强 = 最适合"** | 海外旗舰是真正的能力天花板 | 但比国内贵 5-10 倍，日常任务上 GLM-5.2 / Qwen3-Coder 已够用。我们尊重它的上限，只是中等任务下成本不划算 |

| **反模式 3："benchmark 高 = 好"** | SWE-bench Verified 80% ≠ 你的代码库 80% 成功率 | Pro 才更接近现实，且国内简单题已打平 |

### 如何基于Harness构建结构化文档体系

你给的是"模糊的愿望"，AI需要的是"精确的规范"——这就是结构化文档的意义。

### 什么是结构化文档？三个核心特征

| 特征 | 说明 |
|------|------|
| **特征1：机器可读** | AI能直接解析和执行，有固定字段和格式，不是模糊的人类叙事。例：WHEN/THEN场景描述 |
| **特征2：格式统一** | 有固定模板和结构，字段名全局一致，同类文档同样长相。例：API Spec标准模板 |
| **特征3：相互引用** | 文档之间有引用关系，形成知识网络。UI Spec引用API Spec，实体字段全局对齐 |

> **类比：** 结构化文档 = 外卖订单。告诉外卖小哥"我要吃好吃的" → 送什么都不知道（模糊愿望）。外卖App下单：菜品+地址+备注（少辣/多勺子） → 精确执行。

> 给人看的文档：可以含糊；给AI看的文档：必须精确——这是本质差异。

### Harness五层结构化文档体系

| 层级 | 定义 | 内容 |
|------|------|------|
| **Product Spec** | 定义"做什么" | 产品目标、功能清单、MVP范围界定，哪些功能是核心 |
| **Design Doc** | 定义"怎么做" | 技术架构、选型决策、模块拆分、ADR记录，为什么这样设计 |
| **API Spec** | 定义"怎么连" | 前后端接口契约、字段/类型/错误码、WHEN/THEN行 |
| **UI Spec** | 定义"长什么样" | 页面结构、交互流程、组件状态、布局规范、与API映射关系 |
| **Exec Plan** | 定义"先做什么" | 开发任务拆分、优先级、依赖关系、Sprint的范围 |

> **四注意：** 精确度 > 完整度 | 可验证性 > 描述性 | 分层引用 > 全量加载 | 文档与代码同步演进

#### /opsx 指令链路：需求驱动开发的四步工作流

```
/opsx:explore
```

| 步骤 | 说明 | 类比 |
|------|------|------|
| **/opsx:explore** | 探索现状，扫描 `docs/` + `knowledge/`，定位已有/缺失/过期，输出"信息版图报告" | 医生全面体检 |
